# План развития Habit Tracker в fullstack-проект

Цель этого документа: дать тебе маршрут, по которому ты сам перепишешь текущий React-проект в полноценное приложение с сервером, PostgreSQL, авторизацией и лидербордом. Здесь намеренно нет готового кода приложения. Есть архитектура, этапы, контрольные вопросы и решения, которые стоит принять руками.

## 1. Краткое ревью текущего состояния

Сейчас проект - это клиентское React-приложение на Vite:

- данные привычек лежат в `localStorage`;
- состояние привычек хранится через React Context;
- у привычки есть `id`, `name`, `completions`;
- отметки выполнения считаются на клиенте по датам;
- серверной части, базы данных, пользователей и авторизации пока нет.

Это хороший старт для обучения, потому что у тебя уже есть понятная доменная модель: пользователь создает привычки, отмечает дни, видит прогресс. Теперь можно постепенно переносить ответственность из браузера в backend.

Главная мысль: не надо сразу строить "идеальный production SaaS". Лучше сделать 4 понятных перехода:

1. Из `localStorage` в API.
2. Из API в PostgreSQL.
3. Из одного общего набора привычек в привычки конкретного пользователя.
4. Из личного трекера в социальные механики: лидерборд, публичный профиль, достижения.

## 2. Что я бы выбрал для обучения

Рекомендуемый учебный стек:

- Frontend: текущий React + Vite + TypeScript.
- Backend: Node.js + Express или Fastify.
- Database: PostgreSQL.
- ORM/query builder: Drizzle ORM.
- Validation: Zod.
- Auth: сначала собственная session-based auth на httpOnly cookie, потом можно посмотреть Better Auth.
- Monorepo: npm workspaces.
- API: сначала REST, потом tRPC отдельным этапом.

Почему так:

- REST лучше для базы: ты явно увидишь HTTP, методы, статусы, cookies, CORS, ошибки, границы клиента и сервера.
- tRPC полезен, но для новичка может слишком рано спрятать важные детали HTTP.
- Drizzle ближе к SQL, чем Prisma, поэтому лучше помогает понять таблицы, связи, миграции и запросы.
- Session auth через cookie лучше для обучения, чем JWT в `localStorage`, потому что ты разберешь реальные основы веб-авторизации.

Мой совет: не бери tRPC на первом проходе. Сделай REST API. Когда приложение заработает полностью, создай отдельную ветку и попробуй заменить часть REST-ручек на tRPC. Тогда ты поймешь не только "как использовать tRPC", но и "какую проблему он решает".

## 3. Монорепо: да, но простое

Да, для такого проекта монорепо с workspaces подходит. Но оно должно быть минимальным.

Рекомендуемая структура:

```txt
react-simplified/
  apps/
    web/
      package.json
      src/
    api/
      package.json
      src/
  packages/
    shared/
      package.json
      src/
  package.json
  package-lock.json
  docker-compose.yml
  .env.example
```

Роли:

- `apps/web` - React-приложение.
- `apps/api` - Node.js backend.
- `packages/shared` - общие TypeScript-типы и Zod-схемы, если они действительно нужны и не мешают учиться.
- корневой `package.json` - workspaces и общие команды.

Важно: не надо сразу тащить Turborepo, Nx или pnpm-workspace, если цель - разобраться. npm workspaces достаточно. По официальной документации npm workspaces позволяют управлять несколькими локальными пакетами из одного корневого проекта и запускать команды в конкретном workspace.

Практический порядок:

1. Создай папку `apps/web`.
2. Перенеси туда текущий Vite-клиент.
3. Убедись, что клиент запускается как раньше.
4. Создай `apps/api`.
5. Добавь минимальный сервер.
6. Только после этого подключай PostgreSQL.

Не начинай с идеальной раскладки файлов. Сначала добейся, чтобы `web` и `api` запускались отдельно.

## 4. Какой backend сделать первым

Начни с простого REST API:

- `GET /health` - проверка, что сервер жив.
- `GET /api/habits` - получить привычки текущего пользователя.
- `POST /api/habits` - создать привычку.
- `DELETE /api/habits/:id` - удалить привычку.
- `POST /api/habits/:id/completions` - отметить день.
- `DELETE /api/habits/:id/completions/:date` - снять отметку.

Сначала можно сделать эти endpoints на временном массиве в памяти сервера. Это учебный промежуточный шаг. Он нужен, чтобы ты отделил React от хранения данных.

Когда frontend начнет работать через `fetch`, но данные еще будут теряться при перезапуске сервера, переходи к PostgreSQL.

Что важно понять на этом этапе:

- React больше не "владеет" данными, он отображает данные с сервера.
- `useEffect` нужен для загрузки данных, но не для всей бизнес-логики.
- Ошибки, loading-состояния и пустые состояния становятся частью UI.
- Нельзя доверять данным с клиента: backend должен проверять входные данные.

## 5. PostgreSQL и схема данных

Начни с четырех таблиц:

```txt
users
  id
  email
  password_hash
  created_at

sessions
  id
  user_id
  expires_at
  created_at

habits
  id
  user_id
  name
  created_at
  archived_at

habit_completions
  id
  habit_id
  completed_on
  created_at
```

Ключевые идеи:

- `users` хранит аккаунты.
- `sessions` хранит активные входы пользователя.
- `habits.user_id` привязывает привычку к владельцу.
- `habit_completions.completed_on` лучше хранить как дату без времени, потому что привычка выполняется за день, а не за конкретную миллисекунду.
- На `habit_completions` стоит сделать уникальность по паре `habit_id + completed_on`, чтобы нельзя было отметить один и тот же день два раза.

Что делать руками:

1. Подними PostgreSQL локально, проще всего через Docker Compose.
2. Создай `.env` для backend с `DATABASE_URL`.
3. Подключи Drizzle.
4. Опиши таблицы.
5. Сделай первую миграцию.
6. Проверь базу через GUI или CLI.
7. Перепиши endpoints так, чтобы они ходили в базу.

Не пропускай SQL. Даже если ORM делает миграции, открывай итоговый SQL-файл и читай, какие таблицы реально создаются.

## 6. Авторизация: как правильно для обучения

Для обучения я бы не начинал с OAuth через Google/GitHub. Сначала сделай email + password:

- регистрация;
- логин;
- logout;
- endpoint `GET /api/me`;
- защита endpoints привычек;
- сессия в httpOnly cookie.

Почему cookie-сессии:

- браузер сам отправляет cookie на backend;
- `httpOnly` cookie нельзя прочитать из JavaScript;
- ты увидишь реальную модель "сервер проверяет сессию и понимает, кто пользователь";
- не придется хранить JWT в `localStorage`.

Минимальные auth endpoints:

- `POST /api/auth/register`;
- `POST /api/auth/login`;
- `POST /api/auth/logout`;
- `GET /api/auth/me`.

Что важно не пропустить:

- пароль хранится только как hash;
- email должен быть уникальным;
- login не должен говорить "такого email нет" отдельно от "неверный пароль";
- cookie должна иметь `httpOnly`, `sameSite`, `secure` в production;
- у session должен быть срок жизни;
- logout удаляет session из базы и чистит cookie.

После того как сам поймешь механику, можно изучить Better Auth. По документации Better Auth - framework-agnostic auth framework для TypeScript с email/password, sessions, rate limiting, plugins и управлением БД. Это полезная библиотека, но я бы подключал ее после собственного учебного варианта, а не до него.

## 7. Как привязать привычки к пользователям

Когда появится `user_id`, все операции с привычками должны работать только в рамках текущего пользователя.

Правило:

> Backend никогда не должен принимать `userId` из тела запроса для обычных пользовательских операций.

Правильная цепочка:

1. Клиент отправляет запрос с cookie.
2. Backend достает session из cookie.
3. Backend находит пользователя по session.
4. Backend выполняет запрос только для этого `user.id`.

Пример логики без кода:

- пользователь удаляет привычку;
- frontend отправляет `habitId`;
- backend проверяет, что привычка с таким `id` принадлежит текущему пользователю;
- если принадлежит - удаляет или архивирует;
- если нет - возвращает 404 или 403.

Для учебного проекта лучше использовать soft delete через `archived_at`, а не физическое удаление. Так ты сможешь позже сделать историю, восстановление и аналитику.

## 8. Лидерборд

Лидерборд лучше добавлять после авторизации и PostgreSQL.

Самый простой лидерборд:

- период: текущая неделя;
- метрика: сколько completion-отметок сделал пользователь;
- показывать топ-10 пользователей;
- показывать текущего пользователя отдельно, даже если он не в топ-10.

Но у такого лидерборда есть проблема: пользователь с 30 привычками почти всегда обгонит пользователя с 3 привычками. Поэтому лучше добавить несколько вариантов:

- `total_completions_this_week` - простая активность;
- `completion_rate_this_week` - процент выполненных привычек;
- `current_streak` - текущая серия дней;
- `longest_streak` - рекордная серия.

Для начала реализуй один простой leaderboard endpoint:

- `GET /api/leaderboard?period=week`;

Потом добавь фильтры:

- `period=week`;
- `period=month`;
- `metric=completions`;
- `metric=rate`;
- `metric=streak`.

Что изучить:

- SQL `GROUP BY`;
- `COUNT`;
- `JOIN`;
- фильтрацию по датам;
- сортировку;
- пагинацию.

## 9. Что еще добавить в проект

Хорошие фичи для обучения, в порядке полезности:

1. Профиль пользователя
   - имя;
   - аватар позже;
   - дата регистрации;
   - публичный/приватный профиль.

2. Архив привычек
   - не удалять привычку навсегда;
   - уметь скрывать завершенные или неактуальные привычки.

3. Цель привычки
   - каждый день;
   - N раз в неделю;
   - только выбранные дни недели.

4. Статистика
   - процент выполнения за неделю;
   - календарь за месяц;
   - longest streak;
   - пропущенные дни.

5. Категории привычек
   - здоровье;
   - учеба;
   - работа;
   - спорт;
   - другое.

6. Заметка к выполнению
   - не просто "сделал", а "сделал + комментарий".

7. Публичная страница пользователя
   - показывать только публичные привычки;
   - можно делиться ссылкой.

8. Достижения
   - первая привычка;
   - 7 дней подряд;
   - 30 выполнений;
   - идеальная неделя.

9. Импорт старых данных из `localStorage`
   - полезно как миграционная задача;
   - frontend читает старый `localStorage` и отправляет данные на backend после логина.

10. Тесты
   - backend endpoints;
   - функции подсчета streak;
   - базовые React-компоненты.

## 10. React: что изучать параллельно

Твой текущий проект уже использует `useState`, `useEffect`, Context и custom hook. Дальше порядок такой:

1. Controlled inputs
   - форма создания привычки;
   - login/register формы;
   - валидация.

2. Derived state
   - не хранить в state то, что можно посчитать из существующих данных;
   - например `doneToday`, `streak`, `dateRange`.

3. `useMemo`
   - использовать только когда расчет реально тяжелый или ломает referential equality;
   - не добавлять везде "на всякий случай".

4. `useCallback`
   - изучить после `useMemo`;
   - нужен в основном при передаче callbacks в мемоизированные компоненты или hooks.

5. Context
   - подходит для auth-состояния и текущего пользователя;
   - плохо подходит для часто меняющихся больших данных, если все приложение начинает перерендериваться.

6. Zustand
   - не обязателен прямо сейчас;
   - можно добавить позже для UI-состояния: фильтры, выбранный период, настройки отображения;
   - не используй Zustand как замену серверной БД.

7. Server state
   - данные с backend - это не обычный local state;
   - позже изучи TanStack Query;
   - он помогает с loading, errors, refetch, cache, optimistic updates.

8. Error boundaries
   - пригодятся, когда приложение станет сложнее.

9. React Router
   - нужен, когда появятся страницы: login, register, dashboard, leaderboard, profile, settings.

## 11. Когда подключать TanStack Query

На первом API-этапе можно использовать обычный `fetch` внутри custom hooks. Это поможет понять механику.

Но когда появятся:

- несколько страниц;
- повторные загрузки;
- optimistic update;
- кэширование;
- refetch после mutation;
- отдельные loading/error состояния;

тогда подключай TanStack Query.

Учебный порядок:

1. Сначала вручную: `useEffect` + `fetch` + `useState`.
2. Потом переписать habits loading на TanStack Query.
3. Потом переписать mutations: add, delete, toggle.

Так ты поймешь, какую проблему решает библиотека.

## 12. Когда подключать tRPC

tRPC стоит изучить, но не первым шагом.

Подключай tRPC, когда ты уже понимаешь:

- что такое HTTP endpoint;
- чем `GET` отличается от `POST`;
- что такое status code;
- где происходит validation;
- как frontend получает ошибку от backend;
- как backend понимает текущего пользователя.

Плюсы tRPC:

- типы API автоматически доходят до клиента;
- меньше ручного дублирования типов;
- удобно в TypeScript-монорепо;
- хорошая связка с Zod и TanStack Query.

Минусы для обучения:

- можно хуже понять обычный HTTP;
- сложнее объяснить API человеку, который не знает tRPC;
- меньше практики с REST-дизайном.

Мой выбор для тебя:

1. Версия 1: REST.
2. Версия 2: REST + TanStack Query.
3. Версия 3: отдельная ветка с tRPC.

## 13. Пошаговый маршрут

### Этап 0. Зафиксировать текущее поведение

Цель: понять, что приложение сейчас умеет.

Сделай руками:

- опиши текущие сущности: habit, completion;
- выпиши текущие user flows;
- проверь edge cases: пустое имя, будущая дата, удаление привычки;
- убери временный `console.log` из `useEffect`, если он был только для обучения.

Контрольные вопросы:

- где сейчас создается `id`?
- где сейчас хранятся даты?
- почему после `JSON.parse` нужны Date-объекты?
- что случится, если открыть приложение в другом браузере?

### Этап 1. Перенести проект в monorepo

Цель: структура готова для frontend и backend.

Сделай руками:

- создай `apps/web`;
- перенеси туда текущий React-клиент;
- создай root `package.json` с workspaces;
- проверь запуск клиента через workspace-команду;
- создай пустой `apps/api`.

Контрольные вопросы:

- где лежит root `node_modules`?
- как установить зависимость только в `web`?
- как установить зависимость только в `api`?
- как запустить команду конкретного workspace?

### Этап 2. Сделать backend без базы

Цель: frontend общается с backend, но backend пока хранит данные в памяти.

Сделай руками:

- подними сервер;
- добавь `/health`;
- добавь CRUD endpoints для habits;
- добавь endpoint для toggle completion;
- подключи frontend к API через `fetch`;
- убери запись новых данных в `localStorage`.

Контрольные вопросы:

- почему данные пропадают после перезапуска backend?
- где теперь должна жить бизнес-логика toggle?
- что frontend должен показывать во время загрузки?
- что делать, если backend вернул ошибку?

### Этап 3. Подключить PostgreSQL

Цель: данные переживают перезапуск приложения.

Сделай руками:

- добавь `docker-compose.yml` для PostgreSQL;
- добавь `.env.example`;
- подключи Drizzle;
- опиши таблицы;
- создай миграцию;
- перепиши endpoints на запросы в БД.

Контрольные вопросы:

- чем таблица отличается от TypeScript-типа?
- зачем нужны migrations?
- где должен генерироваться `id`?
- почему completion лучше хранить отдельной таблицей?

### Этап 4. Добавить auth

Цель: у каждого пользователя свои привычки.

Сделай руками:

- регистрация;
- login;
- logout;
- `GET /api/auth/me`;
- middleware, который достает текущего пользователя;
- защита habit endpoints;
- привязка habits к `user_id`.

Контрольные вопросы:

- почему нельзя хранить пароль как plain text?
- почему cookie должна быть `httpOnly`?
- почему backend не должен принимать `userId` от клиента?
- чем authentication отличается от authorization?

### Этап 5. Добавить страницы

Цель: приложение становится полноценным web app.

Страницы:

- `/login`;
- `/register`;
- `/dashboard`;
- `/leaderboard`;
- `/profile`;
- `/settings`.

Что изучить:

- React Router;
- protected routes;
- layout;
- redirect после login/logout;
- состояние "пользователь еще загружается".

### Этап 6. Добавить leaderboard

Цель: использовать SQL не только для CRUD, но и для аналитики.

Сделай руками:

- endpoint лидерборда;
- SQL-запрос с `JOIN`, `GROUP BY`, `COUNT`;
- UI таблицу;
- фильтр периода;
- отображение текущего пользователя.

Контрольные вопросы:

- почему leaderboard считается на backend?
- что будет при 10000 пользователей?
- как добавить пагинацию?
- какая метрика честнее: total completions или completion rate?

### Этап 7. Улучшить frontend data fetching

Цель: привести работу с серверными данными в порядок.

Сделай руками:

- вынеси API calls в отдельный слой;
- добавь единый формат ошибок;
- перепиши habits на TanStack Query;
- добавь optimistic update для toggle;
- проверь поведение при ошибке сети.

Контрольные вопросы:

- чем server state отличается от local state?
- когда надо refetch?
- когда optimistic update опасен?
- где хранить auth user?

## 14. Учебные правила, чтобы не потеряться

1. Делай по одному большому изменению за раз.
2. После каждого этапа запускай приложение и проверяй руками.
3. Не подключай библиотеку, пока не можешь объяснить, какую боль она снимает.
4. Не прячь SQL полностью за ORM.
5. Не храни auth token в `localStorage`.
6. Не добавляй Zustand только потому, что "его используют".
7. Не смешивай UI state и server state.
8. Пиши короткие заметки после каждого этапа: что понял, что осталось мутным.

## 15. Минимальный Definition of Done

Проект можно считать полноценным учебным fullstack-приложением, когда:

- пользователь может зарегистрироваться;
- пользователь может войти и выйти;
- привычки хранятся в PostgreSQL;
- каждый пользователь видит только свои привычки;
- можно создать, удалить, отметить и снять выполнение привычки;
- после перезапуска браузера и backend данные остаются;
- есть leaderboard;
- есть хотя бы один SQL-запрос с агрегацией;
- есть понятная структура `web` / `api` / `shared`;
- ты можешь словами объяснить путь запроса от клика в React до записи в PostgreSQL.
