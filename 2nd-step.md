Да. Второй этап лучше делать не как “сразу backend с PostgreSQL”, а как отдельный учебный слой: **React больше не хранит данные сам, а ходит за ними на сервер**. Сервер на этом этапе может хранить данные просто в памяти. Это нормально.

**Цель этапа 2**

После этапа 2 у тебя должно быть так:

```txt
browser / React
  |
  | fetch(...)
  v
Node.js API server
  |
  v
temporary in-memory array
```

Пока без базы. Если перезапустил сервер - данные пропали. Это ожидаемо.

## 1. Что должно быть в `apps/api`

Минимально:

```txt
apps/api/
  package.json
  src/
    main.ts
```

Позже добавишь:

```txt
apps/api/
  src/
    main.ts
    routes/
    services/
    types/
```

Но сначала не усложняй.

## 2. Что установить

Для новичка я бы начал с **Express**, потому что по нему больше примеров и он проще концептуально.

В `apps/api` нужны зависимости:

- `express` - HTTP-сервер;
- `cors` - чтобы frontend с Vite мог обращаться к API;
- `zod` - позже для проверки входных данных;
- `tsx` - чтобы запускать TypeScript без отдельной сборки;
- `typescript`;
- `@types/express`;
- `@types/cors`;
- `@types/node`.

Пока можно без Drizzle, без PostgreSQL, без auth.

## 3. Какие scripts нужны в `apps/api/package.json`

Тебе нужен хотя бы dev-скрипт:

```json
"scripts": {
  "dev": "tsx watch src/main.ts"
}
```

И из корня удобно добавить:

```json
"dev:api": "npm run dev --workspace @react-simplified/api",
"dev:web": "npm run dev --workspace @react-simplified/web"
```

А потом можно сделать общий `dev`, который запускает оба процесса, но сначала лучше запускать их в двух терминалах отдельно. Так понятнее, где ошибка: во frontend или backend.

## 4. Первый endpoint: `/health`

Начни не с привычек, а с проверки сервера.

Сервер должен слушать порт, например:

```txt
http://localhost:3000
```

И иметь endpoint:

```txt
GET /health
```

Он должен возвращать что-то простое вроде:

```json
{ "ok": true }
```

Это нужно, чтобы проверить:

- сервер вообще запускается;
- порт правильный;
- запросы доходят;
- браузер или Postman/Thunder Client видит ответ.

Пока `/health` не работает, не надо делать habits.

## 5. Потом endpoints для привычек

Сделай временный массив на сервере. Не база, просто переменная в памяти.

Минимальный API:

```txt
GET    /api/habits
POST   /api/habits
DELETE /api/habits/:id
POST   /api/habits/:id/completions
DELETE /api/habits/:id/completions/:date
```

На этом этапе данные еще не привязаны к пользователю. Все привычки общие. Это нормально, потому что auth будет позже.

## 6. Где должна жить логика toggle

Сейчас у тебя toggle логика живет в React:

```txt
если completion на дату уже есть - убрать
если нет - добавить
```

На втором этапе лучше перенести эту бизнес-логику на backend.

Frontend должен говорить:

```txt
"пользователь нажал на дату у привычки"
```

А backend должен решать:

```txt
"эта дата уже отмечена или нет"
```

Так ты постепенно привыкнешь, что сервер отвечает за данные и правила.

## 7. Как React должен общаться с API

В React тебе надо заменить `localStorage`-источник данных на `fetch`.

Сначала вручную:

- при загрузке приложения сделать `GET /api/habits`;
- сохранить результат в `useState`;
- при добавлении привычки сделать `POST /api/habits`;
- после успешного ответа обновить state;
- при удалении сделать `DELETE`;
- при toggle сделать запрос на сервер и обновить state.

Пока не подключай TanStack Query. Сначала руками прочувствуй:

- loading;
- error;
- successful response;
- refetch;
- обновление local state после ответа сервера.

## 8. CORS и порты

У тебя будет два сервера:

```txt
Vite web: http://localhost:5173
API:      http://localhost:3000
```

Это разные origins, поэтому backend должен разрешить запросы с Vite через CORS.

На этапе без cookies достаточно разрешить origin Vite-приложения.

Позже, когда появится auth через cookie, CORS придется настроить строже: `credentials`, конкретный origin, cookie settings.

## 9. Не делай proxy сразу

Можно настроить Vite proxy, чтобы frontend ходил на `/api`, а Vite прокидывал запросы на backend. Но для обучения я бы сначала явно использовал API URL:

```txt
http://localhost:3000/api/habits
```

Так ты лучше увидишь, что frontend и backend - это два отдельных процесса.

Proxy добавишь позже, когда поймешь механику.

## 10. Как проверить этап 2

До подключения React проверь API отдельно:

1. Запусти backend.
2. Открой `http://localhost:3000/health`.
3. Проверь `GET /api/habits`.
4. Создай привычку через Thunder Client/Postman/REST Client.
5. Еще раз вызови `GET /api/habits`.
6. Убедись, что привычка появилась.
7. Перезапусти backend.
8. Убедись, что привычка пропала.

Последний пункт важен: он доказывает, что сейчас данные только в памяти, и следующий этап с PostgreSQL действительно нужен.

## 11. Что должно получиться в конце этапа

Этап 2 готов, если:

- `apps/api` запускается отдельной командой;
- есть рабочий `/health`;
- есть endpoints для habits;
- React больше не использует `localStorage` как основной источник данных;
- создание, удаление и отметка привычек идут через API;
- после перезапуска API данные пропадают;
- ты понимаешь, почему они пропадают.

Главная мысль: **не пытайся сразу сделать красиво**. Сначала добейся тупой, но понятной цепочки:

```txt
button click -> fetch -> Express route -> memory array -> JSON response -> React state update
```
